# Requesty GPT-5.4 reasoning-summary failure modes

Date: 2026-03-26
Repo: `orgs/open-hax/proxx`
Primary stack inspected: `services/proxx/docker-compose.yml`
Clean repro instance: temporary container on `http://127.0.0.1:8797`

## Goal

Investigate intermittent session interruption associated with `reasoning_summaries` / reasoning-summary handling, and document currently known failure modes.

## Summary

There are at least **three distinct failure classes** in the current system:

1. **Historical live-stack upstream rejection in Requesty**
   - Logged in `services/proxx/data/request-logs.jsonl` as:
     - `providerId: requesty`
     - `model: gpt-5.4`
     - `upstreamMode: responses_passthrough`
     - `upstreamPath: /v1/responses`
     - `status: 400`
     - `error: Unknown parameter: 'reasoning_summary'.`
   - Count observed before clean-room repro work: **25** hits.

2. **Provider-prefix routing mismatch on the noisy compose stack**
   - Incoming `requesty/gpt-5.4*` does **not** currently behave like a first-class provider-forcing prefix.
   - In live tests, `requesty/gpt-5.4` and `requesty/gpt-5.4-nano` were repeatedly logged as:
     - `providerId: openai`
     - `upstreamMode: openai_responses_passthrough`
     - `upstreamPath: /codex/responses`
   - `requesty/gpt-5.4-mini` on `/v1/chat/completions` fell into `local_ollama_chat` and failed with `fetch failed` because `ollama` was not resolvable.

3. **Clean requesty-only repro shows a deterministic nested-summary failure for `gpt-5.4`**
   - In an isolated instance pinned to `requesty`, plain `gpt-5.4` on `/v1/responses` behaved differently depending on payload shape:
     - top-level `reasoning_summary: "auto"` -> **200**
     - nested `reasoning: { summary: "auto", effort: "low" }` -> **400** with upstream log error `Unknown parameter: 'reasoning_summary'.`
   - This strongly suggests that somewhere in the requesty path, nested reasoning summary data is being flattened or rewritten into `reasoning_summary` before upstream acceptance is evaluated.

## Live-stack prefix routing findings

### Findings

Using the compose-stack proxy on `http://127.0.0.1:8789`:

- `POST /v1/responses` with `model=requesty/gpt-5.4` timed out client-side, while request logs showed repeated `openai` attempts rather than `requesty` attempts.
- `POST /v1/responses` with `model=requesty/gpt-5.4-nano` showed the same pattern.
- `POST /v1/chat/completions` with `model=requesty/gpt-5.4-mini` returned `500` with `x-open-hax-upstream-mode: local_ollama_chat` and container logs showed `getaddrinfo ENOTFOUND ollama`.

### Interpretation

Current ingress routing recognizes `openai/`, `openai:`, `ollama/`, `ollama:`, and `factory/*`, but **not** `requesty/` as a first-class incoming routing prefix.

Relevant source:
- `src/lib/provider-routing.ts`
- `src/lib/provider-strategy/fallback.ts`

The Requesty-specific `provider/model` prefixing appears to happen only **after** Requesty has already been selected as a fallback candidate.

## Clean mock-upstream instance

To separate proxy behavior from real Requesty behavior, a second temporary clean instance was started on `http://127.0.0.1:8799` with:

- `UPSTREAM_PROVIDER_ID=requesty`
- `REQUESTY_BASE_URL=http://host.docker.internal:8798/v1`
- a local mock upstream listening on `0.0.0.0:8798`

This mock captured the exact outbound JSON body the proxy emitted.

### Mock findings

For an incoming proxy request with top-level `reasoning_summary`, the proxy sent:

```json
{
  "model": "openai/gpt-5.4",
  "input": "Reply with OK only.",
  "reasoning_summary": "auto",
  "max_output_tokens": 16,
  "stream": false
}
```

For an incoming proxy request with nested `reasoning.summary`, the proxy sent:

```json
{
  "model": "openai/gpt-5.4",
  "input": "Reply with OK only.",
  "reasoning": {
    "summary": "auto",
    "effort": "low"
  },
  "max_output_tokens": 16,
  "stream": false
}
```

Important conclusions from the mock capture:

- the proxy prefixes Requesty-targeted GPT models as `openai/gpt-5.4`
- the proxy does **not** flatten nested `reasoning.summary` into top-level `reasoning_summary` before sending to the Requesty upstream URL
- the real-Requesty `Unknown parameter: 'reasoning_summary'.` response for the nested request therefore appears to come from Requesty or a downstream Requesty-integrated layer, not from proxy-side JSON flattening

Interpretation:
- for GPT-family models, chat-completions is still effectively a Responses-backed route, so the same underlying Requesty eligibility / cooldown conditions apply.

## Known failure modes

### Failure mode A: historical Requesty 400 for `reasoning_summary`

Symptoms:
- sessions intermittently interrupted
- request log shows `requesty` + `/v1/responses` + `gpt-5.4` + `Unknown parameter: 'reasoning_summary'.`

Impact:
- request fails after upstream selection
- may appear intermittent from the caller side

Status:
- **confirmed historically**
- **confirmed again in clean repro**, but specifically when nested `reasoning.summary` is supplied

### Failure mode B: unsupported `requesty/` incoming prefix

Symptoms:
- `requesty/gpt-5.4*` does not reliably force Requesty routing
- may instead route to OpenAI or local Ollama logic

Impact:
- makes live debugging misleading
- can hide the real Requesty-specific reasoning-summary problem behind unrelated failures

Status:
- **confirmed live**

### Failure mode C: accidental Local Ollama routing

Symptoms:
- `x-open-hax-upstream-mode: local_ollama_chat`
- `500 fetch failed`
- container log shows `getaddrinfo ENOTFOUND ollama`

Impact:
- unrelated infrastructure failure can mask model/provider routing bugs

Status:
- **confirmed live** for `requesty/gpt-5.4-mini` on `/v1/chat/completions`

### Failure mode D: Requesty policy rejection for `gpt-5.4-mini`

Symptoms:
- proxy returns `502 upstream_unavailable`
- request log shows Requesty `403 Provider blocked by policy`

Impact:
- model may be unavailable on Requesty regardless of reasoning-summary handling

Status:
- **confirmed in clean requesty-only repro**

### Failure mode E: local cooldown / rate-limit saturation for `gpt-5.4-nano`

Symptoms:
- proxy returns `429 all_keys_rate_limited`
- fresh upstream request log entry may be absent if all accounts are locally cooled down before dispatch

Impact:
- can look like a provider bug when it is actually key-pool exhaustion / cooldown state

Status:
- **confirmed in clean requesty-only repro**

## Best current hypothesis

The most likely model-specific reasoning bug is:

- plain Requesty `gpt-5.4` responses traffic can still fail when the outbound payload contains nested `reasoning.summary`
- the proxy mock capture shows that the proxy preserves nested `reasoning.summary` as nested JSON and does **not** flatten it into `reasoning_summary`
- the real Requesty path nevertheless reports `Unknown parameter: 'reasoning_summary'.`
- meanwhile, top-level `reasoning_summary` can succeed for the same Requesty-targeted `openai/gpt-5.4` request shape

This means the current issue is probably **not just "client sent top-level reasoning_summary"** and probably **not a proxy-side flattening bug in the emitted JSON body**. It appears more likely to be a Requesty-side compatibility or translation bug in their handling of `openai/gpt-5.4` Responses requests.

## Recommended next debugging targets

1. Inspect the exact outbound payload emitted to Requesty for the nested `reasoning.summary` clean repro case.
2. Add a regression test around Requesty + `/v1/responses` + `gpt-5.4` + nested reasoning summary.
3. Add explicit ingress support for `requesty/` if provider-forcing by prefix is desired.
4. Prevent accidental Local Ollama fallback when a provider-looking prefix is present but unsupported.
5. Distinguish upstream-policy rejections (`403`) from transport failures (`502`) more directly in user-facing errors.
