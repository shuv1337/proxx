## Recommended shape

A modular monolith is a good fit here because it breaks the code into independent modules by feature, and each module links only to the services it specifically needs, which reduces complexity and makes refactoring safer.  That maps well to your situation because federation and multi-tenancy are exactly the kinds of pressures that expose weak boundaries long before they force you into microservices. [openrouter](https://openrouter.ai/docs/guides/overview/models)

## Where MVC fits

Think of MVC as the boundary pattern, not the domain pattern. In practice, the controller layer handles transport concerns, request normalization, auth context, and response shaping; the “model” visible to MVC should mostly be request/response DTOs and view models, while the real business structure lives one layer deeper in application and domain modules.

That keeps the mental model familiar from Rails, Django, and Spring, but avoids the common trap where controllers become orchestration hubs and “models” become a junk drawer for everything the system knows.

## Strategy and policy

Your strategy pattern and policy engine should stay exactly where they already add value: inside the decision core. A useful high-level split is:
- Controllers: Accept requests, derive tenant and federation context, invoke one use case.
- Application services: Coordinate one business flow such as route-a-request, evaluate-capabilities, or resolve-fallback.
- Domain modules: Contain provider abstractions, routing strategies, policy evaluation, capability rules, and decision objects.
- Infrastructure adapters: Speak OpenAI-style APIs, Anthropic-style APIs, local models, billing stores, tenant config stores, and observability backends.

That separation lets MVC organize the flow at the edge, while strategy and policy organize the behavior at the center.

## Tenant and federation boundaries

Make multi-tenancy and federation first-class architectural boundaries, not incidental flags passed around the codebase. Multi-tenant guidance commonly treats isolation as an explicit architectural choice rather than an implementation detail, with different isolation models chosen based on risk, scale, and operational needs. [aws.amazon](https://aws.amazon.com/solutions/guidance/multi-tenant-architectures-on-aws/)

A practical consequence is that every use case should run inside a clear execution context: tenant, workspace or account, policy bundle, provider access scope, and federation scope. Multi-tenant modular monolith examples also emphasize tenant context handling and policy-enforced access as part of request execution rather than as scattered checks across unrelated code. [github](https://github.com/PeterKneale/micro-saas)

## Packaging model

If you want this to feel natural in TypeScript without adopting a framework, organize by business capability first and by layer second. The strongest lesson from modular monolith practice is that focused modules and disciplined naming conventions make code easier to find, reason about, and change under pressure. [openrouter](https://openrouter.ai/docs/guides/overview/models)

A high-level package map could look like this:
- routing: request classification, route planning, fallback decisions.
- providers: provider registry, capability metadata, adapter contracts.
- tenancy: tenant context, tenant config, isolation rules, quotas.
- federation: cross-cluster or cross-provider resolution, failover regions, trust boundaries.
- policy: evaluation engine, policy definitions, explainability.
- delivery/http: controllers, DTOs, serializers, auth middleware.
- platform: logging, tracing, caching, secrets, metrics.

The key shift is from “what pattern am I using?” to “what boundary owns this responsibility?” MVC then becomes the entry-point shape, while strategy and policy remain the mechanisms that make routing decisions cleanly inside each bounded area.

Would you rather optimize this around a single deployable modular monolith first, or design the module seams now so federation can become separate services later?
