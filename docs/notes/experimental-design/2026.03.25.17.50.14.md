At the top level, this is an **MVC-ish modular monolith for an LLM router**, where:

- **Controllers** = HTTP/API entrypoints (OpenAI-compatible, admin, federation)  
- **Models** = your router’s own domain data and business rules (tenants, providers, credentials, routing policies, usage, catalogs) plus their persistence mapping  
- **Views** = API responses and UI JSON that present that domain in consistent shapes  

Around that, you have a **routing core** (strategy + policy) and **provider adapters** that hide individual LLM vendors.

***

## System overview

Conceptually, the system is:

- A **multi-tenant, multi-provider LLM gateway** implementing an OpenAI-style API surface.  
- Internally, a **modular monolith** with domain models, services, and a routing core, fronted by controllers and backed by a control plane (DB, config, metrics, federation). [engineersmeetai.substack](https://engineersmeetai.substack.com/p/llm-gateway-control-plane-for-ai)

Everything else—strategy pattern, policy engine, provider schemas—is layered under those basic responsibilities.

***

## Controllers: entrypoints into the app

Controllers are the units that speak HTTP and nothing else. In MVC terms, they:

- Accept an HTTP request (path, method, headers, body, auth). [codecademy](https://www.codecademy.com/article/mvc-architecture-model-view-controller)
- Construct a **RequestContext** and **TenantContext** (who is calling, from where, under which plan).  
- Map raw input into **DTOs** for the service layer.  
- Call application services (not strategies/policies directly).  
- Choose and return a **View** (API response shape, streaming, error mapping).

You’d likely have controllers such as:

- `ChatController` – `/v1/chat/completions`, `/v1/responses`, `/v1/messages`.  
- `ModelsController` – `/v1/models`, `/v1/models/:id`.  
- `AuthController` – OAuth/device flows, token refresh, login/logout.  
- `TenantController` – tenant API keys, limits, plan management.  
- `AdminController` – provider/credential/admin operations, health.  

All follow the same pattern: parse, build context, call a use case, format response.

***

## Models: domain + persistence

The **Model** in MVC is your router’s domain and its mapping to storage, not LLMs themselves. [en.wikipedia](https://en.wikipedia.org/wiki/Model_view_controller)

The main domain models are things like:

- **Tenant** – id, plan, quotas, regions, feature flags.  
- **Provider** – id, base URLs, supported capabilities, pricing hints.  
- **Credential/Account** – provider account, plan type, auth type, health, rate-limit state.  
- **ModelCatalogEntry** – logical model id, aliases, provider-specific details, scale/quality signals.  
- **RoutingPolicy** – rules for provider preference, fallback behavior, strategy preference, plan constraints.  
- **RequestUsage / AnalyticsEvent** – per-request usage, errors, cost, latency, labels.

Each has:

- A domain representation (TypeScript types/entities).  
- A **repository** or data-access layer to read/write from DB, env, or config files.  
- Business rules (e.g., “enterprise tenants may use provider X in region Y,” “free plan cannot use certain families”). [blog.logrocket](https://blog.logrocket.com/node-js-project-architecture-best-practices/)

Controllers and services talk to **Models** through repositories and domain services, not directly to the database.

***

## Views: API responses and UI JSON

In this system, “views” are not server-rendered HTML but:

- JSON responses for the OpenAI-compatible API (chat completions, models list, errors).  
- SSE streams for streamed completions.  
- JSON for the operator UI (credentials, tenants, analytics, host dashboard). [developer.mozilla](https://developer.mozilla.org/en-US/docs/Glossary/MVC)

A View layer is responsible for:

- Taking domain objects (e.g., `RoutingResult`, `UsageRecord`, `Tenant`) and mapping them to specific wire formats.  
- Owning compatibility quirks: mapping internal domain data to OpenAI/Anthropic/Gemini/OpenRouter response shapes.  
- Ensuring error responses are consistent (status codes, error bodies).

This is where you keep the “how do we talk to clients?” concern, separate from “how do we decide where to route?” and “what does a tenant look like?”

***

## Application services / use cases

Between Controllers and the core, you have **services** that represent use cases:

- `RouteChatCompletionService` – given a request DTO + context, chooses routes and executes them.  
- `ListModelsService` – reads catalogs and policies, returns logical models.  
- `IssueTenantKeyService`, `RotateCredentialsService`, `GetUsageDashboardService`, etc.

They:

- Orchestrate domain operations for one business use case.  
- Call repositories to fetch models (Tenant, Provider, Credential, Policy, Catalog).  
- Call the **routing core** to decide which upstream(s) to use.  
- Return domain-level results that controllers convert into Views.

This is the “Service layer” pattern that sits on top of MVC Models and below Controllers in many modern Node/TS architectures. [w3tutorials](https://www.w3tutorials.net/blog/controller-service-repository-pattern-nodejs/)

## Routing core: Strategy + Policy as domain logic

Inside the “Model” layer (in the sense of domain logic), you have your two strong cores:

1. **Policy Engine (PDP)**  
   - Encodes **RoutingPolicy**, **account ordering rules**, **strategy preferences**, **fallback behavior**, and **plan constraints** in a structured config.  
   - Given domain-level inputs (Tenant, ModelCatalogEntry, AccountInfo), answers questions like:
     - Which providers are allowed/preferred for this logical model?  
     - In what order should we try accounts and strategies?  
     - What fallback/retry behavior applies for this tenant and model?  

2. **Strategy layer (routing strategies)**  
   - Implements the **how**: call ordering, retry behavior, interpretation of upstream errors, streaming vs non-streaming, etc.  
   - Operates on a `RoutingContext` built from domain models + request DTO:
     - Tenant, requested model, routed logical model, capability needs, health metrics.  
   - Produces a `RoutingPlan` (sequence of provider/account/strategy attempts) and then executes it.

These two together are the core of the router’s *domain logic*. They sit under your services, fed by domain models, and are decoupled from HTTP and from raw provider APIs.

***

## Provider & protocol adapters (integration layer)

Below the routing core is the **integration layer**:

- One adapter per provider family or protocol:
  - OpenAI-compatible HTTP.  
  - Anthropic-style `messages` / `responses`.  
  - Gemini, Ollama, local inference endpoints, etc.  
- Each adapter implements a stable interface:
  - Build an upstream request from an internal request.  
  - Execute it using credentials and config.  
  - Map upstream responses to the router’s internal response shape.

These are **not MVC Models**; they’re infrastructure concerns. They hide provider-specific schemas behind a uniform internal schema, so the rest of the system doesn’t know or care what provider you added.

***

## Control plane: configuration, tenants, analytics, federation

Finally, there’s the **control plane** – still part of the “Model” side in MVC terms, but background/management rather than hot-path:

- **Config & Catalog management** – model catalogs, provider configs, routing profiles, feature flags.  
- **Tenant & credential management** – tenant entities, plans, API keys, provider credentials, OAuth accounts.  
- **Analytics & health** – per-request usage, latency/error metrics, account health, routing decision logs.  
- **Federation & shared state** – how multiple router instances share tenants, credentials, and analytics via a DB or other shared store. [truefoundry](https://www.truefoundry.com/blog/what-is-llm-router)

These are exposed via their own Models/Repositories and sometimes their own Controllers (admin APIs and UI backend), but they are not directly on the chat request hot-path.

***

## How it all fits together (end-to-end)

For a typical `/v1/chat/completions` request:

1. **Controller** (ChatController)
   - Validates the HTTP request, authenticates the caller, builds TenantContext + RequestContext and a ChatRequestDTO.

2. **Service** (RouteChatCompletionService)
   - Loads Tenant, Provider, Credential, ModelCatalog, and RoutingPolicy models via repositories.  
   - Builds a `RoutingContext` and calls the **routing core**.

3. **Routing core**
   - Policy engine orders providers/accounts/strategies based on domain models and rules.  
   - Strategy executes the RoutingPlan: calling provider adapters, handling retries and fallbacks, emitting a unified internal response/stream.

4. **View**
   - Maps the internal response into the OpenAI-compatible JSON or SSE format, adds any required headers, and writes it to the HTTP response.

5. **Control plane**
   - In the background (or as side effects), usage is recorded, health metrics updated, decision logs written, etc., via domain models and repositories.

In this framing, **OpenAI/Claude/Gemini/etc. are just details hidden in the integration layer**, while MVC’s **Model** is your router’s own domain: tenants, providers, credentials, policies, catalogs, and usage, with strategy/policy as the core domain logic sitting on top of that.
