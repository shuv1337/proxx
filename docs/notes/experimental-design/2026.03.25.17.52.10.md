
## Top‑level structure

```text
src/
  app/
    http/
      routes.ts           # all route <-> controller wiring
      server.ts           # Fastify/Express bootstrap
    config/
      env.ts
      config.ts
  features/
    routing/
    tenants/
    providers/
    auth/
    catalog/
    billing/
    analytics/
    admin/
  shared/
    infra/
    core/
    utils/
```

- `app/http` is where framework details live (Fastify/Express/Nest, etc.).  
- `features/*` are your domains (MVC “Models” + Controllers + Services per feature).  
- `shared/` holds cross‑cutting infra (DB, logging, tracing, message bus, etc.), similar to “infrastructure” layer in clean architecture patterns. [blog.bitsrc](https://blog.bitsrc.io/a-clean-and-adaptive-nodejs-architecture-with-typescript-b144c1735447)

## Per‑feature MVC-ish layout

Within each feature, you keep a **consistent internal structure**, so “add a feature” always looks the same:

```text

    ChatController.ts
    ResponsesController.ts
  services/
    RouteChatCompletionService.ts
    RouteResponsesService.ts
```

```text
    Tenant.ts
    TenantPlan.ts
```

Same shape for `catalog/`, `auth/`, `analytics/`, etc. This is very close to what modern “feature‑based” Node/TS architectures recommend: group by domain, and inside each domain keep controllers / services / models / infra consistent. ## Where controllers actually live

Controllers here are framework‑agnostic classes or functions:

```ts
// src/features/routing/controllers/ChatController.ts
export class ChatController {
  constructor(private readonly routeChat: RouteChatCompletionService) {}

  async postChatCompletions(req, res) {
    const ctx = buildRequestContext(req);  // tenant, auth, trace, etc.
    const dto = parseChatRequestDto(req.body);
    const result = await this.routeChat.execute(ctx, dto);
    return writeChatResponse(res, result);
  }
}
```

- **Only place touching `req`/`res`.**  
- Always call a service and then a response mapper.  
- No direct DB calls, no provider calls, no policy logic.

`app/http/routes.ts` then wires Fastify (or whatever) to controller methods and that’s it.

## Services: use cases, not plumbing

Services are “do one thing in the domain” units. Example:

```ts
// src/features/routing/services/RouteChatCompletionService.ts
export class RouteChatCompletionService {
  constructor(
    private readonly routingCore: RoutingCore,           // strategy + policy + execution
    private readonly tenantRepo: TenantRepository,
    private readonly catalog: ModelCatalogService,
  ) {}

  async execute(ctx: RequestContext, dto: ChatRequestDto): Promise<ChatRoutingResult> {
    const tenant = await this.tenantRepo.findById(ctx.tenantId);
    const models = await this.catalog.getModelsForTenant(tenant);
    const plan = await this.routingCore.buildAndExecutePlan({ ctx, dto, tenant, models });
    return plan.result;
  }
}
```

- Services use **repositories** to load MVC Models (Tenant, Provider, Catalog).  
- They delegate to the routing core for decision logic.  
- They return domain‑level results; no HTTP or provider details here.

## Routing core: plug‑and‑play Strategy + Policy

You already have the good parts; we just encapsulate them:

```text
src/features/routing/core/
  StrategyInterfaces.ts     # ProviderStrategy, StrategyRequestContext, etc.
  StrategyRegistry.ts       # register per-provider strategies
  PolicyEngine.ts           # wraps config & engine
  RoutingCore.ts            # build+execute plan
  adapters/
    OpenAIAdapter.ts
    AnthropicAdapter.ts
    GeminiAdapter.ts
    OllamaAdapter.ts
```

`RoutingCore` is what services call:

```ts
// src/features/routing/core/RoutingCore.ts
export class RoutingCore {
  constructor(
    private readonly policyEngine: PolicyEngine,
    private readonly strategyRegistry: StrategyRegistry,
    private readonly providerAdapters: ProviderAdapterRegistry,
  ) {}

  async buildAndExecutePlan(input: RoutingInput): Promise<RoutingPlanExecution> {
    const policyView = this.policyEngine.viewFor(input);
    const strategy = this.strategyRegistry.selectStrategy(policyView);
    const plan = strategy.buildPlan(policyView);
    return this.executePlan(plan);
  }

  private async executePlan(plan: RoutingPlan): Promise<RoutingPlanExecution> {
    // iterate providers/accounts according to plan,
    // call appropriate ProviderAdapter, apply fallback policy, etc.
  }
}
```

To add a new provider or “language model family”:

1. Implement a `ProviderAdapter` in `core/adapters/*`.  
2. Register it in `ProviderAdapterRegistry`.  
3. Optionally add a new `ProviderStrategy` if the call semantics need something special.  
4. Add or adjust policy rules (config) if needed.

No controller changes, no service changes, no random imports—just defined extension points.

## Control plane modules

Each of these is its own feature, with the same internal structure:

- `features/catalog` – models & repos for model catalogs, pricing, aliases.  
- `features/auth` – models & repos for tokens, OAuth state, tenant membership.  
- `features/analytics` – models & repos for usage, latency, cost, decision logs.  
- `features/admin` – controllers/services for operator UI, host dashboard.  

They **do not** sit in the hot path; they back the services that run on the hot path with data and rules.

## Migration mindset vs. greenfield

You’re not going to refactor proxx or your own MVP into this in a weekend. Practically:

1. **Introduce controllers + services around the existing core**  
   - Wrap the current Fastify handlers into controller classes while leaving internals intact.  
   - Extract orchestration into services so handlers become thin.

2. **Identify and factor out domain models**  
   - Start with Tenant, Provider, Credential, RoutingPolicy.  
   - Introduce repositories as facades over the existing DB/files.

3. **Encapsulate the routing core**  
   - Put current Strategy + Policy bits behind a `RoutingCore` façade.  
   - Move provider-specific compatibility code into adapters.

4. **Gradually move into feature directories**  
   - As you touch code, move it under `features/routing`, `features/tenants`, etc., matching the structure above.
