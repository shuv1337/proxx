# PLAN: Claude setup-token auth for Anthropic in proxx

## Goal

Add a **Claude setup-token** authentication option to `proxx` so users can paste the long-lived token generated by:

```bash
claude setup-token
```

and use it as an Anthropic credential inside `proxx`.

This plan is intentionally scoped to:

- ingesting and validating a pasted setup-token
- persisting it safely in the existing credential store
- surfacing it in the existing Credentials UI
- making it usable by the existing Anthropic provider flow
- handling Anthropic quota behavior correctly and transparently
- adding telemetry and tests for the new path

This plan does **not** replace the existing Anthropic browser OAuth code-paste flow. The two auth methods should coexist.

---

## Executive summary

`proxx` already has most of the Anthropic foundation needed:

- Anthropic OAuth manager exists: `src/lib/anthropic-oauth.ts`
- Anthropic quota fetcher exists: `src/lib/anthropic-quota.ts`
- Anthropic refresh dispatch already exists: `src/app.ts`
- Anthropic UI routes already exist: `src/lib/ui-routes.ts`
- Anthropic UI already exists: `web/src/pages/CredentialsPage.tsx`

The recommended implementation is:

1. add a **new paste-only Anthropic setup-token endpoint**
2. validate the token using the same heuristics OpenClaw uses
3. store the token as an **`oauth_bearer` credential with no refresh token**
4. expose it in the credentials UI as a distinct Anthropic auth option
5. preserve the existing browser OAuth path
6. treat Anthropic quota support for setup-token as **best effort**, because OpenClaw documents that some setup-tokens do not have the scope required for the Anthropic OAuth usage endpoint

## Why this design

## Key finding from codebase research

The lowest-risk path is **not** to invent a third auth type like `setup_token`.

Instead, store setup-token credentials as existing `oauth_bearer` accounts with:

- `token = <setup-token>`
- `refreshToken = undefined`
- `expiresAt = undefined`

That fits the current `proxx` architecture because:

- upstream auth already sends credentials as bearer tokens in `src/lib/proxy.ts`
- Anthropic quota currently only includes `oauth_bearer` accounts in `src/lib/anthropic-quota.ts`
- token refresh is already optional in `src/app.ts` and `src/lib/anthropic-quota.ts`
- the credential stores already support OAuth accounts with missing refresh/expiry metadata

## Why not store setup-token as `api_key`

That would be technically possible for raw inference transport, but it is the worse fit because:

- setup-token is used as a **Bearer token**, not a normal Anthropic console API key
- current Anthropic quota logic filters to `oauth_bearer` accounts
- auth-failure handling and UI semantics already align better with OAuth-like bearer credentials than console API keys
- adding a separate auth type would create unnecessary schema, UI, and request-routing churn

## Recommended design

## Credential model

Represent setup-token accounts as:

```ts
{
  providerId: "anthropic",
  accountId: <user-supplied or generated>,
  token: <setup-token>,
  authType: "oauth_bearer",
  refreshToken: undefined,
  expiresAt: undefined,
  email: undefined,
  subject: undefined,
  planType: undefined,
}
```

### Rationale

- matches existing bearer transport behavior
- matches Anthropic quota account inclusion behavior
- requires no new schema or auth type
- lets the UI and backend reuse current Anthropic account rendering paths

## Validation strategy

Adopt OpenClaw’s heuristics first:

```ts
prefix: sk-ant-oat01-
min length: 80
```

Do **not** overfit beyond that initially. Validation should be strict enough to catch obvious mistakes but not pretend to cryptographically verify the token.

## Account naming strategy

Because setup-token does not come with immediate metadata, the add flow should support:

- optional `accountId`
- if omitted, backend-generated fallback using a collision-resistant pattern: `claude-setup-<8-char-random-hex>`
  - avoids collisions when multiple tokens are added in rapid succession
  - timestamp-based IDs are insufficient because sub-second adds would collide

Optional future enhancement:

- support an explicit label field separate from `accountId`

For the first pass, reusing current `accountId` semantics is fine.

## Credential distinguishability

After save, setup-token accounts are **indistinguishable from browser OAuth accounts in the credential store**. There is no stored `credentialKind` or `source` field.

The UI can differentiate them heuristically by the **absence of `email`, `subject`, and `expiresAt`** — browser OAuth accounts will always have these, setup-token accounts will not.

The `credentialKind: "setup_token"` field in the POST response is **response-only** — it exists so the UI can show appropriate post-save messaging but is not persisted.

This is an acceptable v1 trade-off. If future work requires stored differentiation (e.g., to show "setup-token" badges in the account list), a `source` metadata field can be added then.

## Quota strategy

Treat quota for setup-token as **best effort**.

If the Anthropic usage endpoint rejects the token because of missing scope:

- do not fail credential creation
- allow the account to remain usable for inference
- surface quota fetch failure in the Anthropic quota card
- add explicit explanatory copy in UI and/or route error messaging

## UI strategy

Anthropic auth section should expose two choices:

1. **Sign in with Anthropic**
   - existing browser/code-paste OAuth path
2. **Paste Claude setup-token**
   - new direct paste path

Recommended copy:

- “Generate this with `claude setup-token` on any machine, then paste it here.”
- “Some setup-tokens may not support quota visibility; inference can still work.”

## Telemetry strategy

Add structured logs/spans for:

- setup-token validation failure
- setup-token account saved
- setup-token account save rejected
- quota fetch 403 with probable missing scope for setup-token-like credential

Use the existing telemetry style in:

- `src/lib/anthropic-oauth.ts`
- `src/lib/anthropic-quota.ts`

## Risks and mitigations

## Risk 1 — Setup-token saves successfully but quota fails

### Why
OpenClaw documents that some setup-tokens may not have `user:profile` scope required by Anthropic usage APIs.

### Mitigation
- save credential anyway
- show explanatory quota message
- do not treat quota failure as credential-add failure

## Risk 2 — Ambiguous auth type semantics

### Why
Setup-token is not a classic OAuth browser session and not a console API key.

### Mitigation
Use `oauth_bearer` internally because it matches bearer transport and current Anthropic quota logic. Distinguish it in UI copy, not storage schema.

## Risk 3 — Duplicate or confusing account labels

### Why
Setup-token flow does not immediately provide email/subject metadata.

### Mitigation
- accept optional account ID
- use a collision-resistant fallback: `claude-setup-<8-char-random-hex>`
- improve copy so users know they can rename/re-add later if needed

## Risk 5 — Dedup-by-token silently replaces accounts

### Why
`upsertOAuthAccount()` in `credential-store.ts:598-604` filters by both `accountId` and `token`. Re-pasting the same token with a different account ID silently replaces the old entry.

### Mitigation
- this is intentional and matches existing OAuth upsert semantics
- test this behavior explicitly in Phase 2 route tests
- acceptable: one token = one credential record

## Risk 4 — Logging secrets accidentally

### Why
Paste flows often tempt debug logging.

### Mitigation
- never log raw token
- only log provider/account identifiers and validation outcomes
